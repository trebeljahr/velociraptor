# Raptor Runner v1.1.2 — Polish, Pixels, and Pipes

A maintenance cycle. No flashy new feature, but a lot of the rough edges that were getting flagged in playtests got sanded down, and the game quietly started reporting back what players actually do.

## Gameplay feel

- **Mid-flight pterodactyl is fair again.** The low-flight variant used to sit at coin-height, which meant the wings-up frames of its flap cycle were punishing well-timed jumps — you'd clear the body, clip the wingtip, die. Dropped the low band from 0.35 to 0.18 raptor-heights and shrunk the low-flight sprite to 0.8× so the silhouette reads as a nimbler, easier-to-clear flyer. The high-flight run-under variant is unchanged.
- **Diamond got a real sprite.** The end-of-chain gem was hand-drawn with two facets, a gradient, and a moving highlight. It read OK but always felt a little "designed by the same person who wrote the physics code." Ran a Freepik illustration (juicy_fish, blue from the four-colour set) through the sprite-tools CLI — background removed, 2×2 grid sliced, top-left cell trimmed, Lanczos-downscaled to 320×250 — and dropped it in. Then added 10 edge sparkles orbiting the silhouette at 0.7× width so the gem reads as flashier than a regular coin, each one on its own phase offset to avoid flicker-in-lockstep.
- **Cursor is finally crisp.** The custom pointer went through three passes: the outline colour was a medium brown that fought the rest of the UI chrome, then the anti-alias from an upscale+downscale made it fuzzy, then the no-AA version read as too jagged. The shipping version is pure two-colour RGB with smooth alpha-channel AA only on the outer silhouette — outline stays solid `#2a1d13`, edges don't step-stair against the desktop.

## Audio

Two long-standing first-trigger lag spikes got fixed, and the tally on the game-over card no longer bleeds into the next run.

- **Rain no longer stalls the game loop on first trigger.** rain.mp3 is ~3MB, and even with `preload="auto"` Chromium defers the decode until the first `play()`. The previous priming trick played at vol 0 for a moment and then paused — which decoded the first chunk but not the rest, so the first real rain window still hitched as the decoder chewed through the rest. Now the element keeps looping silently after priming, so the full file is decoded well before startRain()'s volume ramp makes it audible.
- **Diamond pickup SFX doesn't spike the frame anymore.** `_coinBuffer`, `_coinChainEndBuffer`, `_shopPurchaseBuffer`, and `_achievementBuffer` were missing from the Web Audio pre-warm list. First diamond grab was compiling their render graphs on the hot path — visible as a single-frame hitch. All four are pre-compiled at unlock now.
- **Game-over tally stops on restart.** The coin-tally was firing up to 10 scheduled rising-pitch chimes plus a final chord via `setTimeout`, and each one spawned a fresh BufferSource. Quick-restarts left ghost chimes ringing out under the fresh run's gameplay audio. The tally's timers and gain nodes now live on the audio module, and `audio.stopCoinFill()` (called from `resetGame`) cancels pending ticks and fades live ones to zero over 80ms.
- **Pop suppression.** Coin pickup, tally ticks, and the chain-end chord now start their gain at 0 and ramp to target over 5ms instead of snapping. Kills the occasional click Chromium emits on BufferSource playback against a non-zero initial sample amplitude.

## Telemetry

New: the game now reports cookieless gameplay events to a self-hosted Plausible instance so I can actually see what players do — run count, score distribution, achievement completion, shop activity — without cookies, without a consent banner, without identifiers.

The whole transport is one file (`src/telemetry.ts`): a single POST to `/api/event` with `keepalive:true` so tab-close events still land. The `shouldTrack()` gate no-ops in Vite dev, Electron (`window.electronAPI.isDesktop`), Capacitor (`window.Capacitor`), and on any host other than the production domain — so desktop installs and preview builds can't pollute the dashboard with un-geolocatable events.

Events fired:
- `pageview` — once per session at module init
- `run_start` — every `initRunState` (first run + every reset)
- `run_end` — in `commitRunScore`, with `{score, coins, nights, jumps, duration_ms, new_best}`
- `achievement_unlock` — with `{id}`
- `shop_purchase` — with `{id}`

Plausible's Events API is identifier-less by design (daily-hashed IP+UA server-side), which keeps the whole thing out of GDPR/ePrivacy consent territory. CSP gets `https://plausible.trebeljahr.com` added to `connect-src`; no inline script changes, so the pinned sha256 hashes still verify clean.

## Under the hood

- **Biome formatter sweep.** A repo-wide format pass landed on main; the worktree in flight during that time had to reconcile by dropping the duplicate local format commit during rebase. No semantic changes, but imports are now alphabetical and line-lengths more consistent across the codebase.
- **`npm` → `pnpm` migration.** Marked separately on main in `e5d07a5`, with native build scripts (electron, sharp, esbuild) approved.

## What's next

Telemetry will be quiet for a day or two while it gathers enough data points to say anything useful, but the obvious first questions it'll answer: what run length do most players finish at, and which achievements are actually reachable without a FAQ. If there's a cluster of players who bounce before their first jump, that's a clearer signal than any playtest feedback has given me.
